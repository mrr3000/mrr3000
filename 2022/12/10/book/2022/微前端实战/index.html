微前端实践

2022-12-10 书籍, 学习, 微前端

一、什么是微前端

优点

  • 优化功能开发：一个团队拥有开发某功能必要的所有技能，不必在独立的前端和后端团队间相互协调
  • 简化前端升级、本地决策：团队可自行决定是否升级或切换他们的技术栈
  • 更专注于客户：每个团队直接向客户提供他们的功能，不存在纯粹的API 团队或运维团队
  • 不再有前端巨石架构
  • 适应变化
  • 遗留项目
  • 减少技术开销、无共享

缺点

  • 冗余：不同技术栈引入大量冗余
  • 一致性：每个团队都是独立
  • 异质性
  • 更多的前端代码

概览图

  • 系统和团队：团队使命、跨职能团队、
  • 前端：页面归属、片段
  • 前端集成：路由和页面跳转、组合、通信
  • 公共：性能、设计系统、共享知识

四、服务端组合

4.1 通过nginx和服务端包含SSI进行组合 server-side includes

  • 服务端组合介于浏览器和实际应用服务器之间的服务来执行
  • 当页面到达客户端浏览器是，已经完全组装完毕，加载速度由于客户端方案
  • 片段组合发生在服务端
  • NGINX 中使用它的服务包含SSI功能来集成
    • 基于url路径转发请求
    • 获取并集成片段

4.2 应对不可靠的片段，超时和回退机制

  • 响应超时：就忽略这个片段。配置一个超时时间 proxy_read_timeout，nginx会将upstream归类为无法工作的upstream，并在10秒内停止连接。但是nginx超时无法做到仅终止个别零星的长时间运行的请求，会将整个upstream标记为无法工作。
  • 回退机制：超时后忽略机制，内容并没有被完全移除，而是在其位置显示了一个show recommendations片段。SSI命令中有一个参数stub,他可以定义一个块的引用，当nginx出错时，nginx会使用这个块的内容。在block endblock 定义回退内容

4.3 深入研究标签的组装性能

  • 并行加载每一个片段，页面标签生成的时间+最慢片段的时间 = 完整的响应时间
  • 支持嵌套SSI includes指令
  • 延迟加载：通过JS使用AJAX获取其内容
  • 优化加速页面加载的技术：
    • 1）首字节时间：它读取主文档，然后等待所有引用片段到达，当页面组装完成后，发送响应给客户端。如何优化？nginx 可以在第一个片段到达之前发送页面模版的头部内容，然后等待所有片段到达后再发送剩下的数据块。分块发送提高性能。
    • 2）流式模版，upstream以流的形式生成和发送标签。

4.4 其他解决方案

  • Edge-side includes: ESI 是一项规范，统一的标签组装方式
  • Zarando Tailor 一个JS库
  • Podium 借鉴了Tailor的思想改进，片段被称为podlet 页面被称为layout

总结

  • SSI ESI 已经过时，但他们集成非常稳定、单调、易于理解
  • 其他库比较封装度高，但也带来了使应用内部的复杂性
  • 优点：
    • 由于浏览器收到的是已经组装完成的页面，因此能够获得出色的首屏加载的性能。
    • SSL ESL 运行可靠且快速，不需要维护太多。
    • 有利于搜索引擎和获得排名
  • 缺点
    • 大型应用，会花费大量时间构建首字节时间，有必要时需要和客户端组合
    • 无法在浏览器中确保技术性隔离。css. Js
    • 根据选择的集成技术，本地开发会变得复杂

五、客户端组合

5.1 使用web component封装微前端

  • 自定义元素：引入了一组生命周期方法 constructor. ConnectedCallback. DisconnectedCallback. AttributeChangedCallback
  • Shadow DOM : 将DOM子树与页面的其余部分隔离开来，消除样式泄漏的可能，从而增强微前端应用的健壮性。 C SS Modules. CSS in js. 都是如此
    • 优点
      • 具有I frame般强大隔离性 不需要命名空间
      • 防止全局样式泄漏，非常适合在遗留应用中使用
      • 减少了对CSS工具链的需求
      • 片段是自包含的，不需要单独的CSS文件引用
    • 缺点
      • 不支持旧的浏览器
      • 需要JS才能工作
      • 无法使用渐进式增强和服务端渲染
      • 很难在不同应用共享通用样式
  • HTML模版

七、客户端路由与应用程序容器

页面刷新：当切换页面时，浏览器需要从服务器加载一个完整的HTML页面。

路由跳转：当切换页面时，完全由客户端重新渲染，通常是使用客户端路由。

7.1 应用程序容器中的扁平化路由

  • 应用程序容器是一个父级应用程序的角色，所有请求都会优先到达应用程序容器，代码会被共享，不包含任何业务逻辑
  • 构建步骤
    • 提供一个共享 的HTML文件
    • 映射URL至对应的团队页面（客户端路由）
    • 渲染匹配的页面
    • 在导航时初始化（销毁）上一页/下一页

7.2 双层路由的应用程序容器

  • 实现一级路由
  • 应用程序容器API包括
    • 加载团队的应用程序代码
    • 基于URL在团队间重定向
    • 上下文信息：语言、国家、用户
    • 元数据处理：更新标签、爬虫提示、语义数据
    • 身份验证
    • POLYFILLS
    • 数据分析与标签管理
    • JS错误报告
    • 性能监控
  • 元框架引入：single-spa 有一些额外功能 JS懒加载和异常处理、框架适配器等，但有一些挑战如下
    • 共享 的HTML文件和元数据
    • 异常边界
    • 内存管理
    • 单点失败
    • 应用程序容器的所有权
    • 通信
    • 引导时间

十一、性能

11.1 高性能架构设计

  • 多团队协作时的性能预算
    • 需要为一个特定的指标定义一个具体的预算，比如大小不超过1m
    • 持续关注性能指标，通过一些工具，lighthouse\ sitespeed.io\ speedcurve \caliber \google analytics
    • 如果出现性能问题，首先：停止开发、排查具体原因
    • 如何解决？
      • 1、将性能预算拆解到微前端的每一个部分
      • 2、页面负责人制
  • 排查性能下降的原因
    • 使用工具进行检查
    • 监控：调试分布式系统，获取以后能用来比较性能的录制文件、网络图、各种指标
    • 将问题隔离：开发者工具中 block url 可以阻止其他团队加载问题
  • 性能收益
    • 天然的代码分割：随着http2流行，将js css分割更小的文件将成为现实
    • 提高缓存能力：仅下载更改的代码部分
    • 减少长时间运行的任务
    • 按需加载
    • 根据用户用例进行优化
    • 快速变更

11.2 精简、复用 vendor库

  • 精简代码
  • 版本管理 we pack dllplugin
  • 中心化 es 模块。rolllup.js
  • 不共享业务代码

十二、UI设计系统

12.1 益处

  • 一致性
  • 通用术语
  • 开发速度
  • 规模化

12.2 公用设计系统与自治团队

  • 有必要搭建自己的设计系统
  • 不断完善设计系统
  • 持续投入、责任到人
    • 不断丰富组件
    • 审视现有的设计
    • 重构
    • 完善文档
    • 提升一致性
  • 获得团队认可
    • 前期准备
    • 让步
    • 沟通
  • 开发流程：集中模式、联合模式
  • 开发阶段

12.3 运行时整合、构建时整合

  • 运行时整合：每个团队引用一个全局css文件，这个文件是官方维护的
    • 这样团队的自治性会被破坏：无法进行沙盒测试；单点故障；无法使用tree shaking；破坏性修改；很难做恰当的版本控制和命名空间。
  • 构建时整合：可以使用版本包来管理：独立升级、剔除未使用的代码、自包含 添加前缀
    • 缺陷：存在冗余、新样式推广慢、一致性、

12.4 样式库中的组件：通用与定制

  • 组件类型的选择
    • 样式
    • 模版技术：服务端渲染还是客户端渲染
    • 行为
  • 改变：拥抱改变、保持简单

12.5 哪些组件应该沉淀到中心化的样式库中

  • 成本：依赖外部项目、可能会被其他团队使用、必须符合更高的质量标准、需要代码审查
  • 组件的复杂度
  • 可重复性
  • 特定领域的组件
  • 团队互信

十三、团队及职责边界

13.1 将系统与团队对齐

  • 明确团队边界
    • 领域驱动设计 ddd：将系统拆分为更小的子模块来处理复杂度，而这些子模块又有明确的关联关系
    • 以用户为中心的设计原则
    • 参考现有的页面结构
  • 团队组成
    • 仅前端团队：规模化开发、方便重构
    • 全栈团队（前后运维）：更具有创造力、降低协作成本
    • 完全自治（前后韵味、商业专家）：快速验证想法、快速适应市场变化
  • 文化改变：垂直的组织架构与以用户为中心的文化相得益彰。大型传统组织很难转垂直组织架构

13.2 知识分享

  • 实践社区community of practice. COP:是一种在团队之间传播知识的方式，由一群拥有相同手艺活着职业技能的人组成。
  • guild可以来自不同团队，有共同兴趣、职业的成员创建一个交流空间。类似slack群组，定期碰面交流。
  • 学习与赋能
  • 展示工作

13.3 横向共性问题

  • 中心化的基础设施：版本控制、持续部署、分析、面板、监控、异常跟踪、托管、公用服务等
    • 软件即服务SAAS
    • 产品开发团队负责建设基础设施
    • 专门的基础设施团队
  • 专业化的组件团队
  • 一致性的协议和约定

13.4 技术多样性

  • 工具箱和默认选择：wiki Java. Vue. We pack. MySQL
  • 前端蓝图
    • 技术细节
      • 目录结构
      • 测试 单元测试、端到端测试
      • 代码检查以及格式化规则
      • 代码风格
